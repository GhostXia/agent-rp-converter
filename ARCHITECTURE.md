# 架构设计：怎么拆，为什么

> ⚠️ **机制更正（优先读 [docs/05](docs/05-loading-mechanism.md)）**：本文早期版本说三份各自"每轮命中 / 常驻"。这是错的 —— Skill 没有"钉住 / 每轮自动注入"。正文是**被调用那一次加载、之后留在上下文**。
>
> 修正后：架构仍是**三份独立 skill**（character / preset / worldbook），但"统一引导"由**元启动器开场显式调用**实现，不是 pin、不是每轮自动。下文"拆 vs 捆"的 token/复用论证依然成立。

## 核心问题

如果角色卡 + 预设 + 世界书捆成一个 Skill，会浪费 token（世界书在闲聊轮也被灌入）、毁掉复用粒度（换文风得重写整份）。所以拆三份。各自独立后，再用一个元启动器在开场把它们统一拉起。

---

## 渐进披露是什么

Skill 系统的设计理念：
- **平时只常驻「名字 + 描述」**（Token 占用极少，几乎为零）
- **用到时才把 `SKILL.md` 正文载入**（按需激活，精准触发）

这意味着：
- 元数据（name+description）一直在，正文只在**被调用那一次**进上下文，之后留存到压缩。
- "统一引导"靠**元启动器开场显式调用**三个 skill（不是 pin、不是每轮自动触发）。
- 世界书做成 skill 懒加载：开场由元启动器调一次（在那之前不占 token）。注意纯 skill **做不到**"提到关键词才加载、用完又卸载"的逐轮门控，只是"调用那一次进上下文然后留着"。
- 若需"每轮必在"的强保证（长会话防漂移），可选地把角色卡+预设额外写进 CLAUDE.md（非默认，见 [docs/05 §2](docs/05-loading-mechanism.md)）。

---

## 如果捆成一个 Skill

```
[巨型 SKILL.md]
├── 角色卡（500 tokens）
├── 预设（200 tokens）
├── 世界书（800 tokens）
└── 总计：1500+ tokens，每轮全量灌入
```

**后果**：
1. Token 浪费：世界书知识在闲聊轮次完全没必要出现
2. 触发失准：所有内容绑定在一起，无法独立复用
3. 维护噩梦：换角色必须重写整份 skill，换世界书必须重写整份 skill
4. 回到原始模式：本质就是"一大段文本塞进系统提示"，没有利用 skill 架构

---

## 正确形态：三份独立 + 元启动器

```
会话开场：/rp-launcher
   ├─→ 调用 skill: character-<name>   → 正文进上下文
   ├─→ 调用 skill: preset-<name>      → 正文进上下文
   └─→ 调用 skill: worldbook-<name>   → 正文进上下文（可选，懒加载）
用户输入 → 模型生成（三件套正文已在上下文）
```

**优势**：
| 维度 | 三份独立 + 元启动器 | 捆成一个巨型单元 |
|------|------|--------|
| Token 效率 | 世界书懒加载，开场前不占 token | 全量灌入，浪费 |
| 复用 | 换世界书不动角色/预设，换文风不动世界 | 一改全改 |
| 维护 | 改角色卡不影响预设 | 牵一发动全身 |
| 统一性 | 元启动器开场一次拉齐 | — |

## 元启动器（Meta Launcher）

如果用户想要"一键启动"，我们不捆内容，而是写一个 **薄壳层（thin shell）**：

```yaml
# rp-launcher/SKILL.md
name: rp-launcher
description: 开始 RP 会话时确认装载就位。不携带内容，只做组合检查。
---
当本 skill 被调用时，确认：
- 指示 agent 调用 character-<name> 与 preset-<name>（拉起角色+文风）。
- 若使用世界书，调用 worldbook-<name>。
- session 干净（无 code/search/data 类 skill 干扰）。
- （可选）若用户配了 CLAUDE.md 常驻升级，确认其在位。
```

这个 meta-skill 正文极短。会话开头调用它一次，做一遍就位检查。它**不能**替你"钉住"任何东西（没这功能），只能提醒你把该放的放好。

## 数据流

```
会话开场：/rp-launcher（一次性拉起）
    ├── 调用 character-<name> → 正文进上下文
    ├── 调用 preset-<name>    → 正文进上下文
    └── 调用 worldbook-<name> → 正文进上下文（可选）
    ↓
用户每轮输入
    ↓
[模型生成]（三件套正文已在上下文）
    ↓
角色化回复（人设在线 + 文风一致 + 知识准确）
```

## 死人化问题的架构解

"死人化" = 编码型 agent 写 RP 会僵硬、机械。

**架构层面不解决**：skill 只能提供静态指令，不能强制模型产生情感。

**但我们最大化缓解**：
1. **角色卡** 里塞 few-shot 对话范例（锚定语气）
2. **预设** 里强制规则："每次回复先以角色身份说一句口语化的内心独白或环境互动，再推进剧情"
3. **世界书** 限定小型，避免角色因知识过载而"全知"
4. **独立 session** 隔离，避免被其他系统提示污染文风

最终效果取决于模型能力，但架构给足了素材。
