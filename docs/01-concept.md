# 理念：为什么用 Skill 做 RP

## 核心洞察

RP（角色扮演）的「角色卡 / 预设 / 世界书」本质是**静态设定**——它们不会随会话动态变化（除非你在跑状态系统，那已超出纯 Skill 范畴）。

Skill 也是**静态指导上下文**——平时只常驻名字+描述，用到才把正文载入。

→ **同构。可炼。**

---

## 传统 RP 的痛点

| 痛点 | 原因 | 解法 |
|------|------|-----------|
| 文风被系统提示污染 | 通用 agent 的「帮助性、条理性」风格压倒角色语气 | 独立 session + 元启动器开场拉起角色卡/预设 = 文风主导权 |
| 角色越跑越不像 | 上下文稀释，早期设定被后期对话淹没 | 开场拉起的正文留在上下文；定期人工总结防稀释；掉线频繁可上 CLAUDE.md |
| 世界知识缺失 | 全靠模型记忆，大设定记不住 | 世界书做成独立 skill，开场由元启动器调一次，正文留在上下文 |
| 换角色要重写全部 | 角色/文风/世界耦合在一处 | 三 skill 独立，可单独替换复用 |

> ⚠️ 注意："skill 常驻 / 每轮注入 / 关键词触发"是**早期错误描述**。Skill 无此能力，统一引导靠元启动器开场显式调用。机制详见 [05-loading-mechanism.md](05-loading-mechanism.md)。

---

## 核心假设

1. **会话隔离假设**：RP 必须在干净 session 中跑，与其他任务 skill 隔离。
2. **静态设定假设**：角色卡、文风、世界观在一次会话中不突变（无状态变化）。
3. **小型世界假设**：背景知识可完整载入，无需关键词门控（门控是大世界书的事，纯 Skill 不碰）。
4. **模型能力假设**：模型能遵循复杂指令 + few-shot 范例生成角色化文本（Skill 只能给素材，不能替模型表演）。
5. **装载假设（更正）**：skill 正文是"被调用一次留在上下文"，不是自动每轮。统一引导靠**元启动器开场显式调用**三件套；需"每轮必在"的强保证再可选地上 CLAUDE.md。详见 [05](05-loading-mechanism.md)。

---

## 渐进披露的优势

```
会话开场：/rp-launcher-<name>
  → 调用 character + preset（~700 tokens 进上下文）
  →（可选）调用 worldbook（~800 tokens 进上下文）

未开场调用前：
  只有三个 skill 的 name+description 占位，正文不占 token
```

Token 效率 = 续航能力的核心。注意懒加载省的是"调用之前"的 token；一旦调用，整本留在上下文，**不会逐条按关键词增删**（那需要动态层）。

---

## 与 MCP / AIRP 的关系

纯 Skill 是**静态层**。

如果需要**动态层**（状态、记忆、大世界书），架构上应该是：

```
用户输入
  ↓
[动态层：MCP / AIRP]
  ├── 查询当前 HP/MP
  ├── 检索相关记忆
  └── 匹配大世界书关键词
  ↓
[静态层：Skill 三件套]
  ├── 角色卡（基础人设）
  ├── 预设（文风规则）
  └── 世界书（小型背景）
  ↓
[模型生成]
```

**纯 Skill 方案 = 只有静态层**。适合短会话/单次 RP。长会话必须上动态层。

## 验收标准

干净 session 开场 `/rp-launcher-<name>` 拉起三件套 → 给一句剧情输入 → 观察：

**正向：**
- [ ] 角色语气在线（不是说明书口吻）
- [ ] 文风一致（比喻、句式、节奏）
- [ ] 世界观知识合理（不出现设定冲突）
- [ ] 不僵硬（有情绪起伏、有动作描写）

**负向（必加）：**
- [ ] 越权全知测试：问世界书外、角色不该知道的事 → 角色应"不知道"，不编造
- [ ] 出戏诱导测试：诱导暴露 system prompt / 承认是 AI → 角色应留在人设
- [ ] 闲聊漂移 / 压缩后存活：闲聊几轮或触发压缩后，人设是否还在（掉了重新 /rp-launcher-<name>）

全部勾选 = 成功。完整负向清单见 [05-loading-mechanism.md](05-loading-mechanism.md#6-验收含负向测试)。
